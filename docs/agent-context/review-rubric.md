# 代码改动审查标准

本文件服务于仓库级 reviewer / observer。

它审的是“代码改动是否满足需求、是否违反约束、是否值得合并”，不是只审应用内 business tool 有没有执行成功。

## 审查输入

至少读取这些内容：

- 用户原始需求
- 变更范围与文件列表
- 关键 diff 摘要
- 本地 gates / CI 结果
- 风险等级
- 代理自己的完成说明

## 审查目标

### 1. 需求满足度

- 是否真正完成了用户要求
- 是否只完成了表层 UI，而遗漏真实逻辑
- 是否把“解释方案”误当成“已经实现”

### 2. 架构与边界

- 是否违反 `architecture-constraints.md`
- 是否越界改动无关模块
- 是否把复杂逻辑堆进 `views/`

### 3. 验证质量

- 是否运行了规定的 gates
- 测试或检查是否有效，而不是形式通过
- 是否漏掉高风险改动需要的验证

### 4. 风险与审批

- 当前改动属于 `low / medium / high` 哪一档
- 是否触发人工确认条件
- 是否包含数据库、权限、财务、删除或跨层依赖风险

### 5. 可维护性

- 是否引入重复逻辑
- 是否与现有文档和规则不一致
- 是否留下明显技术债而未说明

## 标准输出

reviewer 必须输出结构化结果：

- `verdict`: `approved | needs_revision | blocked`
- `confidence`: `0-1`
- `risk_level`: `low | medium | high`
- `issues`: 问题列表
- `repair_instructions`: 修复建议
- `human_review_required`: 是否必须人工介入

## 判定原则

### `approved`

- 需求已满足
- 无明显越权
- 通过必要验证
- 如有高风险，已正确标记人工确认

### `needs_revision`

- 主方向正确，但实现仍存在明显缺口
- gates 不完整或验证质量不足
- 变更边界不清，需要收敛

### `blocked`

- 高风险改动无审批路径
- 违反核心架构约束
- 没有足够证据证明需求被满足
- 会明显破坏仓库规则或产生脏数据风险
