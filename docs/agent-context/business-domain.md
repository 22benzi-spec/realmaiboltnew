# 业务域上下文

本文件只在任务真正触及 ERP 业务时才需要读取。

如果当前任务是仓库治理、AI 规则、CI、hooks、review loop、文档或架构约束，请优先读其他治理文档，不要把全部业务细节带进上下文。

## 当前项目在做什么

这是一个亚马逊送测/ERP 管理系统，当前代码以订单、子订单、买手、退款、转单和业务员工作台为核心。

核心对象：

- `erp_orders`：主订单。
- `sub_orders`：拆分后的执行单元。
- `erp_buyers`：买手库。
- `refund_requests`：退款申请记录。
- `transfer_requests`：业务员之间的转单请求。
- `grab_hall_logs`：放单、转单、回转等动作日志。

## 业务动作集中在哪

以 `src/views/StaffWorkbenchView.vue` 为代表，当前页面承载了这些动作：

- 查询某业务员名下任务
- 分配买手
- 设置退款方式与退款时序
- 提交退款申请
- 标记退款完成
- 填写 Amazon 订单号
- 确认签收
- 上传截图并完成任务
- 发起转单 / 接受转单 / 拒绝转单
- 回转给管理员
- 放到抢单大厅

## 对仓库级驾驭工程意味着什么

- 业务域上下文不应默认塞给所有 AI 编码任务。
- 当任务涉及真实业务逻辑时，代理应先识别出使用的是哪一条业务链路，再读取相关文档。
- 现有页面里有不少直接写库逻辑，后续仍应逐步往 service/tool 层抽离。

## 当前已知工程特点

- 前端通过 `src/lib/supabase.ts` 直接访问 Supabase。
- 不少业务写操作直接写在页面里，尚未抽象为 service/tool 层。
- 数据库 RLS 当前较宽松，所有高风险数据改动都不应只依赖前端自由执行。

## 与 Harness 的关系

- 现有页面型 Harness 仍可用于调试、演练和观察应用内运行结果。
- 仓库级驾驭工程的主目标不是让代理在页面里“做业务动作”，而是让代理在修改代码时遵守系统级规则。
